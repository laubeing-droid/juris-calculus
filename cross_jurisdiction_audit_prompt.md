## 背景

juris-calculus 是一个法域无关的符号法律推理编译器，核心是 Horn 子句 + Fixpoint 不动点迭代。今天（6/13）刚完成了全量 2117 条规则的 DDL（可废止道义逻辑）模态分类。

现在到了岔口：跨法域架构怎么走。

## 当前代码结构中的法域相关部分

```
addons/us/         — 美国法 addon（us_lookup.py 等，骨架阶段）
addons/hk/         — 香港法 addon（hk/rules.yaml 64 条 Horn 规则）
compiler_core/cross_jurisdiction_compare.py  — 跨法域比较引擎（58行骨架）
compiler_core/multi_solver_router.py         — 多求解器路由器（CN/CBL/SPC三轨路由，87行）
compiler_core/de_jure_auditor.py             — De Jure 法定审计器（88行）
configs/zh_CN/rules.yaml                     — 中国法 2,129 条 Horn 规则（主体）
configs/prc_us_alignment/                    — PRC-US 对齐规则目录（60条CBL阻断+23条SPC裁判倾向+131条术语映射，但6/12仓库重构时被暂存删除）
adapter/prc_adapter.py                       — 三轨对撞机适配器（同样在重构时被删）
```

## 核心问题

1. 用户原设计：v1.2.0 已实现三轨对撞（CN + CBL 阻断 + SPC 裁判倾向），但代码在 DDL 升级前的仓库重构中被暂存删除了。今天下午新写的 L1-L2 护栏模块（14 文件 +763 行）在 compiler_core/ 下重新实现了部分法域功能——和旧三轨架构的关系不明确。

2. addons/ 架构已确立——用户的意图是"香港、美国不应该被硬编码在本体，而是做 addons"（用户原话）。用户还想要自动区分法系："比如以后有德国就注册到大陆法"。

3. 用户原始设计是"美国法律界人士来中国也能用，香港做转换层"，而且确实灌入了香港法全文和美国法律词典。但 DDL 做完后用户开始犹豫。

4. 用户正面临的决策（用户原话，按顺序）：
   - "我准备要扩展香港和美国法域，你认为 DDL 是现在这样就好还是等后面我扩完再来"
   - "我本来想做罗塞塔石碑，但是 addon 以后貌似就降低了？"
   - "我原来设计是美国法律界人士来中国也能用，香港做转换层"
   - "美国律师输入美国问题，系统怎么拦截他不出中文结论，我中国律师输入中文要问美国问题，你怎么控制输出结论"
   - "还是我干脆先开发中文版的，以后有需要再跨？"

## 需要独立审计的具体问题

1. **addons 架构 vs 本体硬编码**：当前 addons/us/ 和 addons/hk/ 是独立目录，主体 compiler_core/ 不含法域特定代码。这个架构能否支持未来扩展（+英国/+德国/+日本）而不需要每次改本体？还是应该把跨法域通用逻辑下沉到本体，addons 只放纯数据？

2. **三轨对撞的代码遗产**：v1.2.0 的三轨代码被删了，但今天又重新写了 multi_solver_router.py + cross_jurisdiction_compare.py。这两套代码的关系应该怎么处理？恢复旧代码还是用新的替代？

3. **输出语言控制**：如果系统支持多法域，用户是中文律师输入中文问美国法，输出应该是中文还是英文？如果是美国律师输入英文问中国法呢？这个"N 级复杂化"的问题在编译器层还是前端层解决？

4. **DDL 和法域的关系**：DDL 的模态分类（CONSTITUTIVE/OBLIGATION/PROHIBITION 等）是否跨法域通用？中文法的"应当/不得/可以"和普通法的 shall/must/may 的模态边界是否足够对齐？

5. **"优先中文版"战略的代价**：如果现在决定只做中文版，addons 目录是否需要保留？以后恢复跨法域开发的成本有多高？
